# Architectural review

## Verdict

Your concern is justified. The repository has crossed from **modular** into **atomized**.

The root workspace contains **38 members: 35 packages under `crates/` and three POCs**. Deployable services, public contracts, domain logic, persistence code, test infrastructure, and experiments all appear in one flat namespace. The number itself is not fatal; the problem is that Cargo packages have become the primary way the repository expresses architecture, even when there is no independent deployment, target, versioning, security, or dependency reason for the boundary. The separate `components/` workspace is a different matter: its distinct target and lockfile are justified. ([GitHub][1])

The larger system architecture is also **over-designed relative to the maturity of the core product**. The repository itself acknowledges that the four-tier Postgres topology landed “ahead of the product.” On top of that topology sit a customized wasmCloud runtime, custom WIT capabilities, per-project gateways and runners, scale-to-zero waking, CDC, JetStream, a Postgres run queue, a custom builder, OCI signing, and multiple reconcilers. Each mechanism has a rationale. Collectively, they create a very large correctness, security, and operational proof burden before the basic product loop has been fully validated. ([GitHub][2])

This is a static review of the current `main` branch. I inspected workspace manifests, package dependencies, repository layout, architecture documents, the findings ledger, and selected source hotspots. I did not compile or run the repository.

---

## 1. The worst boundary is `wamn-builder → wamn-host`

`wamn-builder` is an independently deployed build service that processes custom-node source and artifacts, but it depends directly on the production host library to reuse interface screening, grant derivation, and the production engine. That is the wrong dependency direction. It makes the build system inherit the host’s runtime dependency closure and release cadence, while also making the host the owner of policies that are needed outside the host. ([GitHub][3])

The target dependency graph should look like this:

```text
                    ┌──────────────────────┐
                    │ component-policy     │
                    │ import screening     │
                    │ grant derivation     │
                    │ manifest validation  │
                    └──────────┬───────────┘
                               │
                 ┌─────────────┴─────────────┐
                 ▼                           ▼
          wamn-builder                 wamn-node-host
                                             │
                                             ▼
                                      runtime-engine
                                             │
                                             ▼
                                         wamn-host
```

At minimum, extract:

* `wamn-component-policy`: WIT import inspection, allowed-interface policy, grant derivation, egress classification, and manifest validation.
* `wamn-runtime-engine`: only the shared Wasmtime/wash-runtime engine configuration that genuinely must be identical.
* `wamn-node-host`: the custom-node serving process currently embedded in `wamn-host`.

The builder should not depend on the production host. It should depend on policy and, where necessary, a narrowly scoped conformance runtime. This is the highest-value structural refactor because it improves dependency direction, security reviewability, build isolation, and compile times without changing product behavior.

## 2. The repository uses crates where modules would be enough

The house rule says that components should be thin and “logic lives in crates.” That rule has been interpreted too literally. The repository’s own earlier structural review states the better rule: **prefer a module unless target or dependency closure requires a split**. Those two statements need to be reconciled explicitly. Pure core/effect shell is a good design rule; package-per-pure-decision is not. ([GitHub][4])

Packages such as `wamn-impact`, `wamn-migrate`, and `wamn-materializer` are described around a single decision procedure or pipeline. Some may justify independent packages because they must compile into guest components or serve several bounded contexts, but “pure,” “unit-testable,” or “has its own design item” is not sufficient justification by itself. ([GitHub][5])

A new crate should require at least one of these reasons:

1. A different compilation target or ABI, such as native host versus `wasm32-wasip2`.
2. An independently deployed artifact.
3. A public or cross-process contract with its own compatibility policy.
4. A security or dependency boundary that CI actively enforces.
5. A materially different dependency or build profile used by several consumers.

These are **not** adequate reasons:

* The code is pure.
* The concept has its own roadmap item.
* The code might eventually be reused.
* The type has its own design document.
* The module has reached a few hundred lines.

My first consolidation candidate would be the schema family:

```text
wamn-catalog          keep as the canonical contract/model
wamn-schema-core      modules:
                        ddl
                        lifecycle
                        rls
                        seed
                        impact
                        migration
```

That would potentially absorb `wamn-ddl`, `wamn-schema`, `wamn-rls`, `wamn-seed`, `wamn-impact`, and `wamn-migrate`. This should be decided with dependency and change-coupling data, not by blindly merging all six, but they represent one bounded context far more clearly than six root-level peers.

The execution persistence family deserves the same review:

```text
wamn-execution-state
  model
  store
  queue
  reconciliation
  sql
```

`wamn-run-queue` already depends on and composes statements from `wamn-run-store`; that cross-package SQL composition previously produced a parameter-arity defect and led to the creation of `wamn-sql`. A single execution-state package might be clearer, provided it remains guest-compilable. If dependency closure prevents that, retain multiple packages but place them under one explicit `execution/` group. ([GitHub][6])

## 3. The flat namespace is now actively hiding the architecture

The root layout currently treats all these as equivalent:

```text
wamn-host
wamn-catalog
wamn-node-sdk
wamn-gates
wamn-testkit
wamn-builder
wamn-materializer
wamn-impact
wamn-waker
...
```

They are not equivalent. A production process, an external SDK, a frozen wire contract, a pure decision package, and a test harness have completely different architectural roles.

I would reorganize before doing substantial merging:

```text
services/
  host/
  node-host/
  builder/
  dispatcher/
  run-worker/
  cdc-reader/
  waker/
  ctl/

crates/
  contracts/
    catalog/
    flow/
    event-wire/
    node-sdk/
    node-guest/
    node-invoke/
    node-manifest/

  schema/
    core/
    control/

  execution/
    engine/
    state/
    nodes/
    sql/

  event/
    registration/
    materializer/

  runtime/
    component-policy/
    engine/

test-support/
  scenario-model/
  gate-harness/

tests/
  gates/
    local/
    postgres/
    cluster/

poc/
components/
```

This does not initially change package boundaries. It makes the architecture visible in the filesystem and gives CI something meaningful to enforce.

The root workspace should also define `default-members` so normal `cargo build`, `cargo test`, and IDE work do not automatically include POCs and heavyweight cluster gates. POCs can remain workspace members without being part of the default developer loop.

## 4. `wamn-gates` has become an architectural sink

`wamn-gates` depends on the host, control tool, dispatcher, run worker, waker, builder, CDC reader, queue, store, runner, event packages, schema packages, test packages, the runtime fork, Wasmtime, Postgres, and NATS. A system-test package will naturally have a broad graph, but this one has become a second composition root for essentially the entire platform. The repository’s findings ledger also records it as a still-open structural issue: approximately **18.8k lines in a flat package**. ([GitHub][7])

The fix is not “create another crate for every gate.” It is to separate test levels:

```text
Owning package
  unit tests
  property tests
  white-box integration tests

tests/gates/local
  artifact and protocol conformance
  no live infrastructure

tests/gates/postgres
  database behavior
  migrations
  queue/store semantics

tests/gates/cluster
  deployed black-box behavior
  runtime security
  scale-to-zero
  failover
```

White-box gates that instantiate internal engines or call private service functions should move closer to the owning package. The centralized gate suite should increasingly exercise binaries, components, WIT interfaces, HTTP surfaces, database schemas, and observable behavior. Otherwise it prevents internal refactoring because the tests are coupled to every implementation seam.

The `bench`/`proof`/`fixture` vocabulary is internally consistent but nonstandard. At the current size, it needs directory-level grouping and a short taxonomy explaining which tests are correctness gates, performance measurements, fault-injection tests, and demonstrations.

## 5. Production packages depend on something called `testkit`

`wamn-host` has a normal production dependency on `wamn-testkit` because `EgressRecord` lives there. The builder also depends on it for test-case and assertion models. ([GitHub][8])

One of two things is true:

* Those types are real production contracts because customers can supply node/flow tests and the runtime emits observations consumed by them. In that case, `testkit` is the wrong name and it mixes product-domain models with testing machinery.
* They are internal test utilities. In that case, production packages should not depend on them.

I would split this into something like:

```text
wamn-scenario-model
  TestCase
  Assertion
  EgressObservation
  serialized customer-facing/test-runner contracts

test-support/
  evaluators
  fixtures
  fake services
  test-only recorders
```

The same question applies to `wamn-flow-tests`: if stored flow test suites are a product feature, place and name the package as a product contract rather than as repository test support.

## 6. `serve_node.rs` is a process boundary disguised as a source file

`crates/wamn-host/src/serve_node.rs` is **945 lines, 877 lines of code**, and covers custom WIT bindings, credentials and grants, configuration caching, egress policy, Wasmtime instantiation, warm-instance lifecycle, signed-request verification, HTTP serving, command startup, and tests. ([GitHub][9])

That is not primarily a “large file” problem. The file represents a distinct workload:

* It serves one warm custom-node component.
* It has a different threat model from the general host.
* It has its own invocation authentication.
* It has per-node scaling and lifecycle.
* It is emitted as part of custom-node deployment.

That should probably be `services/node-host`, sharing only narrow runtime and policy packages with `wamn-host`.

Even before moving packages, split it internally:

```text
serve_node/
  mod.rs
  auth.rs
  bindings.rs
  credentials.rs
  egress.rs
  instance.rs
  policy.rs
  server.rs
```

The irony in the current repository is that many small domain decisions receive their own crate, while an independently deployable, security-sensitive workload remains one file inside a much heavier crate. That is the wrong granularity in both directions.

## 7. Database schema ownership is still incomplete

The current findings ledger lists two related structural issues as open:

* Two sources of truth for schema.
* The flowrunner reimplements run-state SQL instead of using the owning packages. ([GitHub][10])

The platform has Rust packages emitting SQL while platform tables are also represented in hand-written deployment SQL. This is a more serious architectural problem than the crate count. Package boundaries are only useful if ownership is unambiguous; here, several packages can encode assumptions about the same database schema.

I would establish one of these models:

### Generated schema artifacts

Rust types and migration definitions are authoritative. SQL files are generated and checked into the repository only as build/deployment artifacts, with a drift test.

### Migration files are authoritative

Migration files own physical schema. Rust builds a typed schema manifest from them and tests every SQL builder against the manifest.

The current middle ground—hand-written SQL plus pure string builders plus selective drift tests—will continue producing subtle composition defects. The `wamn-sql` arity type fixes one class, not the ownership problem.

For every platform table, record:

```text
Table
Authoritative schema owner
Migration owner
Read model owner
Write-path owner
Guest-visible SQL owner
Compatibility/version policy
```

No table should have two packages independently authoring statements against undocumented column-order or default assumptions.

## 8. The runtime fork is now part of the product, not a replaceable dependency

The root workspace pins forks of both `wash-runtime` and `pg_walstream`, in addition to pinned Wasmtime dependencies. The findings ledger records that the wash-runtime fork reached five carried commits and had passed the repository’s own escalation threshold for upstream engagement. ([GitHub][1])

This risk is not hypothetical. The runtime’s socket policy previously allowed a critical TCP egress bypass, followed by separate UDP findings. Those issues are recorded as fixed, so they are not current vulnerabilities on `main`; they demonstrate that the security properties of Wamn depend on detailed behavior inside the fork, not merely on WIT declarations in this repository. ([GitHub][10])

The process around the forks appears documented, but the architectural cost remains. I would enforce:

* Wamn business logic never enters the fork.
* Every carried patch has an upstream issue or pull request.
* Every patch has a negative conformance gate in the Wamn repository.
* There is a hard patch-count and age budget.
* Runtime upgrades are exercised continuously, not only when a feature forces one.
* The control plane records the exact runtime and carried-patch set for each deployment.
* A quarterly decision explicitly chooses between upstreaming, maintaining, replacing, or removing each patch.

The custom Postgres capability may be strategically necessary. It should nevertheless be treated as a permanent platform subsystem with its own compatibility and upgrade contract, rather than as glue around wasmCloud.

## 9. The event architecture is coherent, but its proof burden is high

The event path is roughly:

```text
Postgres WAL
    ↓
CDC reader
    ↓
JetStream
    ↓
durable consumer per subscribing flow
    ↓
materializer
    ↓
Postgres run queue
    ↓
NATS doorbell
    ↓
run worker
```

The design includes useful mechanisms: stable event identifiers, deterministic run IDs, durable consumers, conflict-based deduplication, and Postgres as the run-state authority. ([GitHub][11])

However, the documentation calls deterministic `run_id` plus `ON CONFLICT DO NOTHING` an “exactly-once guarantee.” That is too broad. It can provide **exactly-once run creation for one event-flow pair**. It cannot provide end-to-end exactly-once external effects when workers are at-least-once and nodes may call nontransactional external systems. The execution design itself describes at-least-once semantics with idempotency keys. The terminology should be narrowed so operators and customers do not infer stronger guarantees than the system offers. ([GitHub][11])

For each hop, maintain an invariant table:

| Hop                      | Authority                 | Delivery              | Dedupe key             | Replay source     | Failure visibility      |
| ------------------------ | ------------------------- | --------------------- | ---------------------- | ----------------- | ----------------------- |
| WAL → JetStream          | Postgres WAL              | At least once         | Event/message ID       | WAL/slot          | Reader and slot metrics |
| JetStream → run row      | JetStream + PG uniqueness | At least once         | Flow + stream sequence | Durable consumer  | Consumer lag/refusals   |
| Run row → worker         | Postgres queue            | At least once         | Run ID/lease           | Queue row         | Queue age/lease state   |
| Worker → external effect | External system           | Usually at least once | Node idempotency key   | Run history/input | Node-specific evidence  |

The open finding that the CDC replication credential has cluster-wide WAL visibility is also an important reminder: this event design creates a high-privilege subsystem and should be isolated accordingly. ([GitHub][10])

## 10. “Per-project nearly free” is an unproven deployment assumption

The generated API design says one gateway instance per project is “nearly free” because of Wasm density and scale-to-zero. Compute can be cheap while control-plane cardinality remains expensive. Every project can still imply CRDs, routing state, secrets, catalog state, database pools, NATS subscriptions, metrics series, reconciliation work, upgrade work, and cold-start coordination. ([GitHub][12])

The security argument for isolated project instances is reasonable. But these are different questions:

1. Does each project need its own isolated component instance?
2. Does each project need its own pod or host process?
3. Does each project need its own Kubernetes Service and route?
4. Does every inactive project need resident control-plane objects?
5. Must every tier use the same topology?

Do not lock all five together.

Before treating the topology as settled, measure at least:

* 100, 1,000, and 10,000 provisioned projects.
* Mostly idle versus mostly active projects.
* Full-cluster reconciliation time.
* Kubernetes API and etcd object pressure.
* Host memory from cached component state and DB pools.
* NATS consumer and subject counts.
* Rolling-upgrade completion time.
* Cold-start tail latency and thundering-herd behavior.
* Cost per idle project including control-plane services.

The four-tier Postgres work can remain, but no further topology should be added until those measurements and the core vertical product path justify it.

## 11. Standard-node isolation is not a hard security boundary

The architecture is explicit that one flow-runner embeds the standard-node library and receives the union of its required capabilities. Per-node restrictions inside that runner are logical dispatch policy, not cryptographic or process isolation. Custom nodes receive a harder component boundary. ([GitHub][13])

That is acceptable provided the model is stated plainly:

* Standard nodes are trusted platform code.
* A defect in one standard node can potentially exercise the runner’s complete capability set.
* Dispatch-time policy is defense against mistakes, not against hostile standard-node code.
* Risky or third-party nodes belong in separately sandboxed custom components.
* Capability-class runners are an available future split, not something to build prematurely.

The documentation is mostly honest about this already. Product and security materials should preserve the same precision.

## Boundaries that should remain separate

Not every small package should be merged.

The following boundaries have objective reasons:

* `wamn-node-sdk` should remain a minimal leaf.
* `wamn-node-guest` should remain separate so `wit-bindgen` does not enter the SDK dependency closure.
* `wamn-node-invoke` is a shared guest/host wire contract.
* `wamn-event-wire` is a producer/consumer compatibility boundary.
* Native and component workspaces should retain separate targets and lockfiles.
* Builder, CDC reader, host, and likely run worker should remain distinct deployable artifacts because their privilege, scaling, and failure domains differ.
* `wamn-ctl` is a reasonable composition root for one-shot administration and should not be fragmented into one binary per verb. ([GitHub][14])

The repository does not need a monolith. It needs fewer **accidental** package boundaries and more visible **intentional** ones.

## Recommended package decisions

| Area                                                     | Recommendation                                                                                   |
| -------------------------------------------------------- | ------------------------------------------------------------------------------------------------ |
| `node-sdk`, `node-guest`, `node-invoke`, `node-manifest` | Keep separate and group under `contracts/node/`.                                                 |
| `event-wire`                                             | Keep separate as a versioned transport contract.                                                 |
| `catalog`, `flow`                                        | Keep as canonical published/data contracts.                                                      |
| `ddl`, `schema`, `rls`, `seed`, `impact`, `migrate`      | Strong consolidation candidates into a schema bounded context.                                   |
| `run-store`, `run-queue`, `sql`                          | Group immediately; merge store/queue if target closure permits.                                  |
| `event-reg`                                              | Probably keep as a lightweight contract; group under `event/`.                                   |
| `materializer`                                           | Keep separate if guest compilation and reuse justify it; otherwise make it an event-core module. |
| `provision`, `registry`, `sysschema`                     | Review as one control-plane bounded context rather than three root peers.                        |
| `testkit`, `flow-tests`                                  | Decide whether they are product contracts or test support; rename and split accordingly.         |
| `wamn-host::serve_node`                                  | Extract into `node-host`.                                                                        |
| Host import screening and grant derivation               | Extract into `component-policy`.                                                                 |
| `wamn-gates`                                             | Retain a black-box cluster suite, but distribute white-box tests back to owners.                 |

I would expect the native workspace to settle somewhere in the **low-to-mid twenties**, but that should be an outcome rather than a quota.

## Migration sequence

### 1. Classify and move without changing code

Assign every package exactly one role:

```text
service
contract
domain-core
adapter
test-support
system-test
poc
```

Move directories and introduce `default-members`. Do not rename or merge packages in this step.

### 2. Add an architecture check

Use `cargo metadata` in an `xtask` or CI script to enforce rules such as:

```text
contracts     must not depend on services or adapters
domain-core   must not depend on services
services      must not depend on other services
production    must not depend on test-support
components    may depend only on guest-safe contracts/core
builder       must not depend on host
```

Allow explicit, documented exceptions rather than relying on naming conventions.

### 3. Extract `component-policy` and `node-host`

This removes the worst dependency inversion and gives the custom-node serving path an appropriate process boundary.

### 4. Restructure the test system

Move white-box tests to owners, split local/Postgres/cluster gates, and resolve the production dependency on `testkit`.

### 5. Consolidate one bounded context at a time

Start with the schema family. Use:

* dependency closure,
* consumers,
* target requirements,
* Git co-change history,
* release/versioning needs,
* and compile-time impact.

Merge packages that always change and release together. Merely nest the packages that have legitimate independent boundaries.

### 6. Make data ownership explicit

Close the two open schema/SQL ownership findings before adding more persistence packages or database paths.

### 7. Freeze new infrastructure mechanisms

Until the core flow/API vertical slice and project-cardinality tests are convincing, require an explicit removal or simplification plan for every new service, store, protocol, fork patch, or project-scoped Kubernetes object.

---

## Bottom line

The raw number of crates is the visible symptom. The deeper problems are:

* Package boundaries sometimes represent concepts rather than architectural boundaries.
* The flat filesystem hides the difference between services, contracts, cores, and tests.
* The builder depends in the wrong direction on the host.
* A distinct node-host process is buried inside the host.
* Schema and SQL ownership are not fully singular.
* The centralized gate package is becoming a second implementation.
* The runtime adaptation surface is now substantial enough to be a core product subsystem.
* The deployment and data-plane topology carry more operational complexity than the present product proof warrants.

The first concrete refactor should be **extracting `component-policy`, separating `node-host`, and eliminating `builder → host`**. The next should be **filesystem tiering plus dependency rules**, followed by selective schema/execution consolidation.
