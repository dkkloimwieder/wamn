## Headline

The codebase is, on the whole, **high quality**: the pure-core/effect-shell discipline holds almost everywhere, SQL is centralized parameterized text-builders, errors are consistent WIT-mirroring enums, the node contract and generated-API compiler are genuinely well-designed, and the test/gate suite is unusually mutation-resistant. The review found **no reachable critical bug on a shipped default path**. It did find one **critical verification-integrity defect** (a load-bearing gate that cannot fail) and a **coherent architectural theme** that matters more than any single finding:

> **AR1 â€” The security model assumes a trust level the roadmap is about to remove.** Four independent findings (C4-1, C3-1, C1-1, C2-3) share one root: isolation invariants that hold *today only because every SQL statement, credential handle, and node invocation originates from trusted platform code* are **not enforced at the boundary where untrusted input will arrive** â€” the raw-SQL node (D8/`wamn-1nd`), custom-code nodes (5.6/`wamn-bd5`), and multi-author flows. Each is latent now and live the moment those ship. They should be closed **before** those features, not after, and they form the natural prerequisite set for the next roadmap phase.

## Critical

### C7-1 â€” The S1 `bench` gate is print-only; it computes every pass flag and discards all of them
`crates/wamn-gates/src/bench.rs:196-198` computes `p99 < 10ms`, prints `PASS(...): {pass}`, and returns `Ok(())`; `capkill_phase:324` hardcodes `println!("PASS(...): true")`; `epochkill_phase` and the R3 memory-differentiation `membudget_phase:553-595` do the same. `run():141-149` awaits each phase (propagating only I/O errors) and ends `Ok(())` â€” no `bail!`, no aggregation. It is the only print-only module in an otherwise uniformly exit-code-disciplined suite (20/21 modules aggregate `bool` phases and `bail!`), and it has no `deploy/bench-job.yaml`.

**Why critical:** `bench` is the *sole* backing for S1 (cold-instantiation p99), the R3 per-component memory-limiter differentiation gate (64/192 MiB â€” the D16 gate of record), and the R4 fork-upgrade safety net ("run the S1 subset, check exit code" before bumping the wash-runtime fork rev). A regression that breaks the epoch interrupt, defeats the memory limiter, or blows instantiation latency prints `PASS: false` and **exits 0**. The evidence base for the fork's two carried commits is decorative.

**Fix:** phases return `Result<bool>`; aggregate `pass &= â€¦`; `if !pass { bail!(â€¦) }` â€” the shape every other module already uses. Delete the hardcoded `true`. Add a `bench-job.yaml`.

## High

### C4-1 â€” A guest can override its host-injected `app.tenant` claim in-band and defeat RLS tenant isolation *(review headline)*
The plugin docstring (`crates/wamn-host/src/plugins/wamn_postgres.rs:10-11`) asserts "there is no guest-facing way to set or override claims." This is **false at the SQL level.** `HostTransaction::query`/`execute` (`:1365-1405`) pass arbitrary guest `sql` to `run_query`/`run_execute` with zero statement filtering. Tenant identity is injected only as `SET LOCAL app.tenant` at BEGIN (`:656-682`), and RLS keys entirely on `NULLIF(current_setting('app.tenant', true), '')` (`deploy/postgres-init.sql:55`). `app.tenant` is an unreserved placeholder GUC that the `wamn_app` login role (`NOSUPERUSER NOBYPASSRLS`) may freely `SET`; a later `SET app.tenant = 'victim'` in the same transaction overrides the BEGIN-time `SET LOCAL`. A guest on the plugin's own transaction API therefore does `begin()` â†’ `execute("SET app.tenant='victim'", [])` â†’ `query("SELECT * FROM sensitive")` and reads/writes another tenant's rows. S2's RLS gate never attempts an in-band override (`pgbench.rs:485-560` only reads foreign rows via a WHERE pattern), and the 2.6 egress review considers only raw-socket bypass, so this vector is untested and undocumented.

**Reachability:** not reachable on shipped default paths (standard nodes emit structured/parameterized SQL; the raw-SQL node is flag-OFF; custom nodes not shipped) â€” hence **high, not critical** â€” but it is **directly exploitable the moment the raw-SQL node (`wamn-1nd`) is enabled or custom nodes (`wamn-bd5`) ship**, and it silently invalidates the per-tenant-RLS assumption that `wamn-286`'s own scoping analysis leans on. Note the D8 decision-table rationale ("containment is gate-proven â€” RLS 0 leaks â€¦ plugin-only DB path") is materially weaker than stated once this and C7-3 are accounted for; a dedicated user-SQL role (`wamn-1nd`) does **not** close it, because any role can `SET app.tenant`.

**Fix:** (1) add an S2 gate phase that drives the transaction API to `SET`/`set_config('app.tenant', <foreign>)` and asserts zero cross-tenant rows â€” make the invariant *tested*; (2) bind tenant identity to something the guest cannot rewrite in-band â€” a per-tenant DB role via `SET ROLE` with RLS keyed on `current_user`, or a `GUC_SUPERUSER_ONLY` `app.tenant` set through a small extension; (3) correct the docstring and `docs/security-db-path.md` to scope the claim to the config path until (2) lands. **Hard-blocks `wamn-1nd` enablement and `wamn-bd5`.**

### C3-1 â€” The credential vault enforces project scope but never the frozen contract's per-execution `not-granted` grant
`wamn:node@0.1.0` models per-execution grants: `credential-error::not-granted` = "handle exists but was not granted to this node execution" (`docs/wamn-node.wit:181-189`). The host vault implements only *project* scope: `credentials.get(handle)` resolves any handle in the component's project (`wamn_credentials.rs:213-239`), and `resolve` returns only `Unavailable`/`NotFound`, never `not-granted` (`:158-167`). The only node-scoping lives in the SDK `CapsCtx` facade (`wamn-node-guest/src/caps.rs:98-110`), which the direct-import custom-node path bypasses. `project_for` additionally fails **open** to `DEFAULT_PROJECT` on a missing `wamn.project` claim (`:147-154`).

**Impact:** a custom node (the contract's entire reason to exist) importing `wamn:node/credentials` directly can `get(<any-name>)` and receive every credential in its *project's* vault, including secrets declared by sibling flows it was never granted â€” defeating the "secret injected only into the executing node's context" invariant the contract advertises. Within-tenant, latent today (only the compiled-in standard library calls credentials, via the safe facade), live when custom nodes ship.

**Fix:** enforce host-side â€” the runner registers the executing dispatch's declared credential name(s) with the plugin (as it already `set_project`s), and `resolve` returns `not-granted` for any other handle. Non-breaking (the WIT variant exists). Make `project_for` fail-**closed**. **Blocks `wamn-bd5`/`wamn-1ab`.**

### C1-1 â€” Catalog `Check` and RLS `RolePredicate` expressions are spliced verbatim into DDL executed as a multi-statement batch
`emit.rs:255` (`ADD CONSTRAINT {c} CHECK ({expression})`) and `wamn-rls/src/compile.rs:184-186` (`â€¦ OR ({expression})`) insert author-supplied expression text raw; validation checks only non-emptiness (`validate.rs:311-319`, `wamn-rls/src/validate.rs:134-140`); the migrate driver runs the param-less statement via `tx.batch_execute` (`migrate_catalog.rs:159`, simple protocol â†’ multiple `;`-separated statements). A `Check` expression such as `1=1); DROP TABLE app_system.users; --` chains arbitrary statements into the apply transaction at migration-role privilege â€” the build-time analog of the runtime role-scoping the `wamn-1nd`/`wamn-286` beads exist for, currently unfiled and (for `Check`) undocumented as a trust boundary. Blast radius = the migrate connection's grants (PLAUSIBLE; a DDL-capable role can reach `app_system`/`wamn_run` in the same DB).

**Fix:** (1) document catalog/policy authorship as trusted to the migration-role level; (2) apply migrations under a least-privileged DDL role with no `app_system`/`wamn_run` grants (build-time mirror of `wamn-1nd`); (3) defense-in-depth â€” reject expressions with unbalanced parens or a statement terminator so chaining is unrepresentable.

### C2-1 (+ C4-3) â€” A permitted cycle has no termination bound at any layer; one non-terminating run permanently wedges a project's run-worker
`wamn-flow/src/validate.rs:191` rejects only self-loops (`from==to`); longer cycles pass and reachability is a warning. The engine `drive`/`next` (`wamn-runner/src/engine.rs:435,272`) have no step or wall-clock cap. The run-worker deliberately disables the host backstop: `set_epoch_deadline(u64::MAX/2)` "never traps a legitimately long run" (`run_worker.rs:315-317`, = C4-3). The lease renews per node (`run-queue.md:220`) so a live infinite run's lease never expires and the janitor (reaps only *expired* leases) never reclaims it. A tenant loop whose terminating `conditional` never flips â€” the docs explicitly invite loops as "structural" â€” blocks the single-instance run-worker inside one `run-next` forever, renews the lease so failover/janitor cannot reclaim, starves every other run in the project, and issues unbounded node effects (`aâ†’http-requestâ†’a` = unbounded egress). `run-queue.md:266-269`'s drain-termination proof is unsound for a run that neither terminates nor parks.

**Fix:** a run-level guard in the pure engine (max node-executions or max wall-clock on `drive`/`RunState`) surfacing a terminal `FailKind` so the run dequeues and frees the runner; a finite epoch deadline (or `tokio::timeout` on `call_run_next`) as defense-in-depth. Natural home for `wamn-dq5` [5.12] on the runner path.

### C5-1 â€” Paginated list loses row-order stability under a user-supplied `sort` (no unique tiebreaker)
`router.rs:404-408` appends the `"id" ASC` tiebreaker **only** when the user supplied no sort; `?sort=status&limit=50&offset=50` emits `ORDER BY "status" ASC` with no unique final key. On non-unique keys (`status`/`received_at`/`opened_at`) Postgres returns equal-key rows in implementation-defined physical order that can differ between the page-1 and page-2 `OFFSET` queries even against one snapshot â€” so a client paging the inspector hold-queue (`wamn-3n3`) silently **skips or duplicates** rows. A missed quality hold is a real safety miss for this POC's domain. **Fix:** always append `quote_ident("id") ASC` as the final `order_by` element (one line).

### C6-1 â€” The tier-move plan has no quiesce step; an automated saga loses every write made during the dumpâ†’flip window
`plan_tier_move` (`tier_move.rs:98-140`) emits `ProvisionClusters â†’ per-env{Dump, ProvisionEnv, Restore} â†’ FlipRegistry` with no read-only/quiesce/verify step. `resolve()` routes to the OLD cluster until the final `FlipRegistry`, and `databaseReclaimPolicy: retain` keeps the old DB live and writable throughout (`move_org_tier.rs:264-275`). Every write between an env's `Dump` snapshot and the flip lands only in the old DB and is absent from the restored new DB. `postgres-topology.md:375` calls it a "downtime window" but nothing enforces downtime â€” and `wamn-2ib`, whose job is to *drive the plan automatically*, will faithfully drop a paying customer's in-flight writes at cutover. There is also no post-cutover deprovision of the old (retained, tenant-data-bearing) DBs and no row-count verification before flip.

**Fix:** add an explicit `Quiesce{triple}` primitive (revoke writes / `default_transaction_read_only` / logical-replication cutover) + `VerifyRestored`/`DeprovisionOld`; the saga refuses cutover until quiesce + verification are recorded. **Prerequisite for `wamn-2ib` tier-move automation** (see the saga-readiness assessment below).

### C7-2 â€” Wake-doorbell (and pgbench multiproject) soft-skip to PASS under the `--mode all` gate-of-record path when their dependency is absent
`dispatchbench.rs:335` calls `wake_phase(required = args.mode == Mode::Wake)`, so under `Mode::All` (the gate of record â€” `deploy/dispatchbench-job.yaml` runs `--mode all`) `required=false`; a NATS connect failure then prints "(skipping wake gateâ€¦)" and `return Ok(true)`, counted as PASS (`:1044-1050`). Same shape in `pgbench.rs:258-268` (multiproject skips on a missing admin URL). If in-cluster NATS is down/misconfigured, the async-warm doorbell delivery gate (the D15 "async warm p50 < 25ms" / exactly-once-wake evidence) silently becomes a no-op and the Job still exits 0 â€” a false-green vector for a suite that is the project's CI substitute. **Fix:** a `--require-nats` on in-cluster Jobs, or default `required=true` under `Mode::All`.
